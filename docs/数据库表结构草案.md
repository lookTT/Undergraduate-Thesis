# 数据库表结构草案

## 1. 设计原则

- 只保留支撑核心业务闭环的表。
- 以“少表、清晰、易实现”为目标。
- 表结构要方便论文中的需求分析、系统设计和实现说明。

## 2. 核心表设计

### 2.1 用户表 `user`

用途：

- 存储系统登录用户的基础信息。

建议字段：

- `id`
- `username`
- `password_hash`
- `real_name`
- `phone`
- `status`
- `create_time`
- `update_time`

说明：

- 用户表只保存登录与基础身份信息。
- 角色信息不直接写死在用户表中，使用关联表管理。

### 2.2 角色表 `role`

用途：

- 存储系统角色类型。

建议字段：

- `id`
- `role_code`
- `role_name`
- `description`
- `create_time`

建议角色：

- 管理员
- 志愿者

### 2.3 用户角色关联表 `user_role`

用途：

- 建立用户和角色之间的关联。

建议字段：

- `id`
- `user_id`
- `role_id`

说明：

- 当前项目可以先按单角色设计。
- 预留关联表是为了论文结构更规范，也方便后续扩展。

### 2.4 志愿者档案表 `volunteer_profile`

用途：

- 存储志愿者的业务档案信息。

建议字段：

- `id`
- `user_id`
- `student_or_member_no`
- `gender`
- `age`
- `community_name`
- `skill_tag`
- `remark`
- `create_time`
- `update_time`

说明：

- 该表是志愿者业务信息的核心表。
- `skill_tag` 建议先做成文本字段，减少开发量。

### 2.5 志愿活动表 `activity`

用途：

- 存储志愿活动的发布信息。

建议字段：

- `id`
- `title`
- `content`
- `location`
- `start_time`
- `end_time`
- `recruit_count`
- `status`
- `creator_id`
- `create_time`
- `update_time`

建议状态：

- 待发布
- 招募中
- 进行中
- 已结束

### 2.6 活动报名表 `activity_signup`

用途：

- 记录志愿者对活动的报名申请和审核结果。

建议字段：

- `id`
- `activity_id`
- `volunteer_id`
- `apply_time`
- `audit_status`
- `audit_time`
- `audit_remark`

建议状态：

- 待审核
- 已通过
- 已拒绝

### 2.7 活动签到表 `activity_checkin`

用途：

- 记录志愿者参加活动后的签到信息。

建议字段：

- `id`
- `activity_id`
- `volunteer_id`
- `checkin_time`
- `checkin_status`
- `checkin_note`

说明：

- 如果后续想继续简化，也可以把签到信息合并到服务记录表中。
- 当前先单独建表，逻辑更清晰。

### 2.8 服务记录表 `service_record`

用途：

- 记录志愿服务时长统计结果。

建议字段：

- `id`
- `activity_id`
- `volunteer_id`
- `service_hours`
- `record_source`
- `create_time`

说明：

- 服务时长可以在活动结束后统一生成。
- 该表用于论文中的统计功能和结果展示。

## 3. 表关系说明

- `user` 与 `role` 通过 `user_role` 关联。
- `user` 与 `volunteer_profile` 一对一关联。
- `activity` 与 `activity_signup` 一对多关联。
- `activity` 与 `activity_checkin` 一对多关联。
- `activity` 与 `service_record` 一对多关联。

## 4. 建议的最小实现范围

- 先实现 `user`、`volunteer_profile`、`activity`、`activity_signup`、`activity_checkin`、`service_record` 六张核心表。
- `role` 和 `user_role` 可以在权限控制实现时补齐。
- 不新增公告表、日志表等非核心表，避免范围膨胀。

## 5. 后续落地顺序

1. 先建表。
2. 再写实体类和 Mapper。
3. 再实现登录和权限。
4. 再实现志愿者、活动、报名、签到、统计流程。
