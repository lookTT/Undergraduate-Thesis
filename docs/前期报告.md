## 一、研究背景及意义

在社区日常工作中，志愿者的参与越来越重要，从文明宣传到关爱帮扶都离不开他们。但实际管理中，大多数社区仍然依赖纸质登记、微信群发通知、Excel 手动统计这些传统方式。这种做法在人少活动少的时候还能应付，一旦志愿者多了、活动频繁了，问题就出来了：志愿者信息分散在各处，想找个人得翻好几个表格；活动发出去谁报了名、审核到哪一步了，经常搞不清楚；活动结束要统计服务时长，光是核对签到记录就要花大半天。管理人员疲于应付这些琐事，志愿者也常常因为信息不及时而错过参与机会。

一个完整的志愿服务管理流程，应该包括志愿者信息维护、活动发布、报名审核、现场签到和时长统计这几个环节。现在的问题是这些环节分散在不同工具里：志愿者名单在Excel里，活动通知在微信群里，签到靠纸质表格，最后统计又得重新汇总。数据重复录入不说，各个环节之间也衔接不上。做一套把这些功能整合在一起的系统，能切实解决这些痛点。

技术选型上，我选择了目前主流的前后端分离方案。后端用 Spring Boot 3 搭建 REST 接口，处理认证和业务逻辑；前端用 Vue 3 配合 Element Plus 做管理界面；数据存储用 MySQL；安全方面用 JWT 配合 Spring Security。这套技术栈的好处是资料多、遇到问题容易找到解决方案，作为毕业设计项目来说比较稳妥。

做这个课题有两方面的意义。从应用角度看，系统能帮助社区把志愿者和活动的信息统一管理起来，减少人工统计的麻烦，让整个流程有据可查。从学习角度看，这个项目涵盖了需求分析、数据库设计、接口开发、前后端联调等软件开发的完整流程，是一个很好的实践机会。

## 二、国内外研究现状

查阅相关资料后发现，现有的志愿服务管理系统主要围绕志愿者注册、活动发布、报名审核、签到记录、时长统计这些功能展开。国外的平台比如 VolunteerHub、SignUpGenius 更注重多组织协作和开放接口，功能做得比较全面；国内的系统则更贴近具体场景，比如校园志愿平台、社区志愿小程序等，侧重于日常管理和审批流程。

技术实现上，这类系统已经从早期的简单信息登记页面发展成了前后端分离的 Web 平台。后端用 Spring Boot 或 Node.js，前端用 Vue 或 React，数据库用 MySQL，这种搭配在中小型管理系统中很常见，技术成熟度高，开发成本也相对可控。

从现有项目的经验来看，社区志愿者管理系统不需要做得太复杂，关键是把活动发布、报名审核、签到统计这几个最常用的功能做好，保证稳定好用。其他高级功能可以后续再考虑添加。

整理文献时我注意到，Spring Boot + Vue 的组合已经是中小型管理系统的主流选择，RBAC 权限控制也是标配做法。这些都是比较成熟的技术方案。

不过，现有的一些系统功能做得太全太复杂，对于基层社区来说反而不好上手。所以我在设计时决定做减法，只保留最核心的”登录—志愿者管理—活动发布—报名审核—签到—统计”这条主线，把业务流程走通走顺，而不是追求大而全。

## 三、研究的主要内容

本课题的主要工作内容包括：

1. **需求分析与角色划分。** 梳理社区志愿服务的业务流程，明确管理员和志愿者各自要做什么，找出当前管理方式中信息分散、流程混乱、统计困难等痛点。
2. **系统架构设计。** 采用前后端分离架构，前端用 Vue 3 + Element Plus，后端用 Spring Boot 3 + MyBatis-Plus，数据库用 MySQL，整体结构清晰，便于后续维护。
3. **数据库与接口设计。** 设计用户表、角色表、志愿者档案表、活动表、报名表、签到表等核心数据表，理清表之间的关系；同时按照 RESTful 规范设计各业务模块的接口。
4. **核心功能实现。** 实现登录认证、角色权限控制、志愿者档案管理、活动发布、报名审核、签到记录、服务时长统计等功能，把整个业务流程串起来。
5. **测试与优化。** 测试各功能模块是否正常工作，检查权限控制是否到位，处理各种异常情况，修复发现的问题。

简单来说，这个项目不追求功能多么全面，而是要把最核心的业务流程做完整、做稳定，保证系统好用。

## 四、前期工作报告

### 4.1 工作过程

前期工作按照”选题—文献调研—需求分析—架构设计—开发实现—联调测试”的顺序推进。

第一阶段是选题。我选择做社区志愿者服务管理系统，因为这个场景业务明确、功能范围可控，能完整体验软件开发的全过程。

第二阶段是查阅资料。我看了不少关于 Spring Boot、Vue、权限控制方面的文档和文章，了解了当前主流的技术方案，也参考了一些开源项目的做法。

第三阶段是确定需求和技术路线。根据社区志愿服务的实际情况，确定了管理员和志愿者两类角色，围绕”档案管理—活动发布—报名审核—签到—统计”这条主线来设计。技术栈确定为 Spring Boot 3 + Vue 3 + MySQL。

第四阶段是系统设计和基础搭建。画了数据库 ER 图，设计了表结构，搭好了前后端项目框架，配置好了安全认证和异常处理。

第五阶段是功能开发和测试。陆续完成了报名、签到、统计等核心功能的开发，边开发边测试，修复了前后端数据同步和页面交互方面的一些问题。

目前系统的主要功能已经跑通，进入了完善细节和优化体验的阶段。

### 4.2 遇到的问题

开发过程中遇到了几个比较棘手的问题：

1. **状态流转的逻辑控制。** 活动有”未发布”、”报名中”、”待审核”、”签到中”、”已结束”等多种状态，每种状态能做什么操作有限制，比如未审核通过就不能签到，同一个活动不能重复报名。这些校验逻辑如果考虑不周全，很容易出现数据混乱。

2. **前后端数据同步。** 前后端分离后，前端页面上显示的数据和数据库里实际的数据有时不一致，比如刚提交了报名，列表里却没显示出来。需要在操作成功后及时刷新数据。

3. **权限控制和防越权。** 系统里有管理员和普通志愿者两种角色，要防止普通用户通过修改请求参数去操作别人的数据或者执行管理员才能做的操作。

4. **表单交互体验。** 活动发布页面要填的信息比较多，需要做好输入校验和错误提示，让用户知道哪里填错了、该怎么改。

### 4.3 解决问题的方法、效果与启示

针对这些问题，我的解决办法是：

状态流转方面，在写代码前先把各种状态的转换规则理清楚，然后在后端的关键操作前加上校验，不符合条件就直接拒绝。

数据同步方面，统一了接口的字段命名和返回格式，前端在增删改操作成功后自动刷新列表，确保页面显示和数据库一致。

权限控制方面，用 Spring Security 的 `@PreAuthorize` 注解控制接口访问权限，在涉及数据修改的操作中还会额外校验当前用户是否有权操作这条数据。

表单交互方面，用 Element Plus 的表单组件配合校验规则，把常见的校验逻辑封装起来复用，保证各个页面的交互风格统一。

这些措施效果还不错，核心业务流程已经能稳定运行了。我也意识到，开发之前把业务规则和技术方案想清楚，比后期反复修改要高效得多。
