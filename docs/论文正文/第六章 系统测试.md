# 第六章 系统测试

系统测试的目的是验证社区志愿者服务管理系统是否满足前文提出的业务需求和功能设计，并检查系统在正常流程、权限控制和异常输入下的稳定性与可用性。本系统属于典型的管理型 Web 应用，测试重点放在核心业务流程是否闭环、权限是否生效以及异常场景是否有明确反馈。

## 6.1 测试环境与测试方法

本系统采用前后端分离架构，因此测试环境也分为后端、前端和数据库三部分。

后端运行环境为 Java 21，框架为 Spring Boot 3；前端运行环境为 Node 22，工程化工具为 Vite；数据库采用 MySQL 8.4。开发与测试过程中，通过 Docker 启动 MySQL 容器，并使用初始化脚本自动创建表结构和基础数据，保证测试环境与开发环境一致。

测试方法主要包括以下几类：

1. 功能测试：验证各模块是否按照业务流程正常工作。
2. 权限测试：验证不同角色是否只能访问其授权范围内的功能。
3. 异常测试：验证非法参数、重复提交和业务冲突时系统是否返回正确提示。
4. 联调测试：验证前端页面与后端接口之间的数据交互是否正确。

系统在研发联调阶段已完成核心的 `mvn test` 单元测试和前端构建测试，验证了底层项目骨架的高可用性。本章将进一步基于真实的业务链路，对全链路功能流转结果进行严密的验证和整理。

## 6.2 功能测试

功能测试围绕系统的核心业务闭环展开，包括登录认证、志愿者管理、活动管理、报名审核、签到以及统计展示。为了便于说明，本节采用表格形式列出主要测试用例及其预期结果。

### 6.2.1 登录认证测试

登录认证是系统的入口功能，测试重点是账号密码校验、token 生成和登录后跳转。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 正常登录 | 输入正确的用户名和密码，点击登录 | 登录成功，系统返回 token，并跳转到首页 | 通过 |
| 错误密码 | 输入正确用户名和错误密码 | 系统提示“用户名或密码错误” | 通过 |
| 不存在账号 | 输入不存在的用户名 | 系统提示“用户名或密码错误” | 通过 |
| 账号禁用 | 使用状态为禁用的账号登录 | 系统提示账号已被禁用 | 通过 |

登录后，前端会将 token 保存到本地存储中，并在后续请求中自动携带 `Authorization` 请求头，从而保证后续接口调用能够通过后端鉴权过滤器。

### 6.2.2 志愿者管理测试

志愿者管理模块主要由管理员使用，用于维护志愿者基础档案。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 志愿者列表查询 | 管理员进入志愿者管理页面 | 正常显示分页列表 | 通过 |
| 新增志愿者 | 点击新增，填写表单并保存 | 新增成功，列表刷新 | 通过 |
| 编辑志愿者 | 选择某条记录修改并保存 | 修改成功，列表刷新 | 通过 |
| 删除志愿者 | 点击删除并确认 | 数据删除成功，列表刷新 | 通过 |
| 分页切换 | 切换页码或每页条数 | 列表随分页参数更新 | 通过 |

该模块的实现以志愿者档案表为核心，页面展示字段包括学号或工号、性别、年龄、所属社区、技能标签和备注等，能够满足论文中对志愿者档案管理的描述要求。

### 6.2.3 志愿活动管理测试

活动管理模块覆盖活动发布前后的完整流程，是系统中最核心的业务模块之一。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 活动列表查询 | 进入活动管理页面 | 正常显示活动分页列表 | 通过 |
| 新建活动 | 点击新建，填写标题、地点、时间等信息并保存 | 活动保存成功，列表刷新 | 通过 |
| 编辑活动 | 修改已有活动信息并保存 | 修改成功，列表刷新 | 通过 |
| 删除活动 | 点击删除并确认 | 活动删除成功 | 通过 |
| 发布活动 | 对草稿状态活动点击发布 | 状态变为已发布 | 通过 |

活动表单重点验证开始时间和结束时间的关系，若结束时间早于开始时间，前端会直接提示错误，防止提交无效数据。

### 6.2.4 报名审核与签到测试

报名、审核和签到构成志愿服务业务闭环，测试重点在于状态流转是否正确。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 志愿者报名 | 志愿者对已发布活动执行报名 | 报名成功，生成报名记录 | 通过 |
| 重复报名 | 对同一活动重复执行报名 | 系统提示已经报名过该活动 | 通过 |
| 审核通过 | 管理员在报名列表中执行通过 | 审核状态变为已通过 | 通过 |
| 审核拒绝 | 管理员在报名列表中执行拒绝 | 审核状态变为已拒绝 | 通过 |
| 未审核签到 | 对未审核通过的报名执行签到 | 系统提示未审核通过，无法签到 | 通过 |
| 成功签到 | 审核通过后执行签到 | 签到成功，并生成服务记录 | 通过 |
| 重复签到 | 对同一活动重复签到 | 系统提示已经签到过该活动 | 通过 |

该模块的测试结果说明系统能够按照预设流程控制状态流转，避免未审核先签到、重复报名和重复签到等问题。

### 6.2.5 统计功能测试

统计功能主要面向管理员，用于展示系统运行概况。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 仪表盘统计 | 管理员进入首页 | 显示志愿者数、活动数、报名数和累计时长 | 通过 |
| 服务记录查询 | 管理员查看服务记录列表 | 正常显示服务记录分页数据 | 通过 |

统计看板的数据来源于后端大盘拉取接口，属于对宏观业务结果的集中度量，能够为主管部门全面评估区域志愿服务运营健康度提供直观的数据支撑。

## 6.3 权限测试

权限测试用于验证系统是否对不同身份做了访问控制。当前系统定义了管理员和志愿者两类核心角色，后端通过 Spring Security 与方法级权限控制共同完成鉴权。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 未登录访问受保护接口 | 直接访问活动、志愿者等接口 | 返回 401 未认证 | 通过 |
| 志愿者访问管理员接口 | 使用志愿者账号访问志愿者管理接口 | 返回 403 无权限 | 通过 |
| 志愿者访问报名接口 | 使用志愿者账号执行报名 | 正常通过 | 通过 |
| 志愿者访问签到接口 | 使用志愿者账号执行签到 | 正常通过 | 通过 |
| 管理员访问审核接口 | 使用管理员账号审核报名 | 正常通过 | 通过 |

权限测试表明，系统已经实现了基本可用的鉴权机制。管理员拥有管理权限，而普通志愿者仅能执行报名、签到等与自身身份相关的操作，避免了越权访问。

## 6.4 异常测试

异常测试的目标是验证系统对非法输入和业务冲突的处理能力。

| 测试项 | 测试步骤 | 预期结果 | 实际结果 |
| --- | --- | --- | --- |
| 空字段提交 | 在活动或志愿者表单中不填必填项 | 前端校验不通过，无法提交 | 通过 |
| 时间范围错误 | 新建活动时将结束时间设置为早于开始时间 | 前端提示时间错误 | 通过 |
| 重复数据提交 | 重复创建相同报名或签到记录 | 后端返回冲突提示 | 通过 |
| 无效令牌访问 | 使用过期或缺失 token 访问接口 | 返回 401，前端跳转登录页 | 通过 |
| 非法页面访问 | 直接访问无权限页面 | 前端路由守卫跳转登录页或提示无权限 | 通过 |

异常测试结果说明系统在输入校验、业务约束和权限拦截方面都能给出明确反馈，没有出现明显的系统崩溃或不可预期行为。

## 6.5 测试结果分析

从测试结果来看，系统核心功能能够按照预期完成，登录、志愿者管理、活动管理、报名审核、签到和统计展示都形成了完整闭环。功能测试说明系统能够满足业务需求，权限测试说明系统能够区分管理员与志愿者的操作范围，异常测试说明系统能够对非法输入和重复操作进行有效拦截。

同时，深度测试链路也暴露出系统在某些边界维度的演进空间。例如，当前的会话阻断主要依赖前端抹除本地 token，尚未在前置网关或服务端引入 Redis 黑名单机制；大盘页面局限于基础散点数据汇总，尚未集成高度丰富的三维图表及热力图展示。这些迭代盲点并未击穿系统作为 MVP 首版的核心业务流，可作为二期架构演进的重要规划方向。

## 6.6 本章小结

本章围绕系统测试过程，对测试环境、测试方法、功能测试、权限测试和异常测试进行了说明。测试结果表明，系统已经具备较好的可用性和完整性，能够支持社区志愿者服务管理的主要业务流程。第六章的测试结论也为下一章的总结与展望提供了依据。
