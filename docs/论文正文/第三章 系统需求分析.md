# 第三章 系统需求分析

社区志愿者服务管理系统面向社区日常志愿服务场景，主要解决志愿者信息分散、活动发布与报名流程不统一、签到和时长统计依赖人工整理等问题。本章从业务场景出发，对系统的角色需求、功能需求、非功能需求以及可行性进行分析，为后续系统设计与系统实现提供依据。

## 3.1 业务需求分析

在社区志愿服务管理过程中，管理人员通常需要完成志愿者档案维护、活动发布、报名审核、签到确认和服务时长统计等工作。如果仍采用纸质登记、Excel 表格或即时通讯工具进行管理，往往会出现以下问题：

1. 志愿者信息分散，难以统一维护。
2. 活动报名依赖人工统计，容易出现重复报名和遗漏。
3. 签到记录缺乏统一格式，后期核对困难。
4. 服务时长需要人工汇总，统计效率低且容易出错。
5. 活动状态、报名状态和签到状态缺少统一流转机制，不利于管理追踪。

针对上述问题，系统需要提供一个统一的信息化管理平台，使志愿者能够便捷查看和报名活动，管理员能够高效完成活动发布、报名审核和统计汇总，从而形成“登录认证 - 档案管理 - 活动发布 - 报名审核 - 签到确认 - 服务统计”的完整业务闭环。

结合敏捷开发理论与实际社区落地情况，本文将系统首期迭代目标控制在核心业务的最小可用闭环（MVP）上，暂不引入复杂的多级审批、外部系统对接或跨组织协同等非核心功能，以确保系统能够快速交付、验证并投入实际应用，降低前期的研发与试错成本。

## 3.2 角色需求分析

本系统主要包含管理员和志愿者两类核心角色。

### 3.2.1 管理员需求

管理员是系统的主要管理者，负责维护系统中的基础数据和业务流程。管理员的主要需求如下：

1. 登录系统并获取管理权限。
2. 维护志愿者档案信息，包括新增、修改、删除和分页查询。
3. 创建志愿活动，设置活动标题、时间、地点和招募人数。
4. 发布活动，使活动对志愿者可见。
5. 查看活动报名列表，并对报名结果进行审核通过或拒绝。
6. 查看活动签到记录，确认志愿者到场情况。
7. 查看统计概览，包括志愿者数量、活动数量、报名数量和累计服务时长。

管理员角色强调管理与审核职责，因此系统必须对相关接口做权限控制，防止普通志愿者访问管理功能。

### 3.2.2 志愿者需求

志愿者是系统的业务参与者，主要参与活动报名和现场签到。志愿者的主要需求如下：

1. 使用账号密码登录系统。
2. 查看已发布的志愿活动列表和活动详情。
3. 对感兴趣的活动执行报名操作。
4. 在活动审核通过后进行签到确认。
5. 退出系统并清理登录状态。

志愿者只需要访问与自身参与流程相关的功能，不能访问志愿者管理、活动审核和统计汇总等管理类接口。

## 3.3 功能需求分析

结合业务场景和角色划分，系统的功能需求可以归纳为以下几个方面。

### 3.3.1 登录认证需求

系统应支持用户使用用户名和密码登录；登录成功后系统返回 token 和当前用户信息，并在前端保存登录状态。用户退出时，前端应清理本地会话并返回登录页。系统还应能够根据当前登录用户的角色进行权限判断。

### 3.3.2 志愿者管理需求

管理员应能够对志愿者档案进行统一管理，包括列表查询、详情查看、新增、编辑和删除。志愿者档案应至少包含学号或工号、性别、年龄、所属社区、技能标签和备注等字段，以便满足后续活动报名和统计需求。

### 3.3.3 志愿活动管理需求

管理员应能够创建并维护志愿活动信息，活动信息至少应包含标题、内容、地点、开始时间、结束时间和招募人数。活动在草稿状态下可编辑，发布后志愿者可见；管理员应支持活动列表查询、详情查看、编辑、删除和发布操作。

### 3.3.4 报名审核需求

志愿者在看到已发布活动后，应能够发起报名申请。管理员应能够查看活动报名列表，对报名记录执行通过或拒绝操作。系统应支持报名状态管理，并避免同一志愿者对同一活动重复报名。

### 3.3.5 签到与服务统计需求

当志愿者报名通过后，应能够对活动进行签到确认。系统应在签到成功时生成签到记录，并自动生成对应的服务记录，用于后续统计服务时长。管理员应能够查看服务记录和统计概览，了解系统中的志愿者数量、活动数量、报名数量和累计服务时长。

### 3.3.6 核心功能聚焦与 MVP 策略

针对社区治理的实际落地场景，系统采用 MVP（最小可行性产品）策略聚焦最优先的核心业务闭环。为防止首期产品功能过于臃肿而导致交付延期或偏离主线业务，系统暂不扩展弱关联的复杂消息中心、多层级跨部门审批流等功能。部分辅助模块（如密码修改等配置项）作为预留扩展接口存在，以此保证核心流程的研发质量与稳定性。

## 3.4 非功能需求分析

除了基本业务功能外，系统还需要满足一定的非功能性要求。

### 3.4.1 安全性需求

系统应具备基本的身份认证和权限控制能力，只有登录用户才能访问受保护接口，只有管理员才能访问志愿者管理、活动审核和统计等管理类功能。系统应采用 JWT 和 Spring Security 组合实现基本鉴权，降低未授权访问风险。

### 3.4.2 易用性需求

系统界面应简洁清晰，常用功能应能够通过少量点击完成。前端页面应以表格、弹窗和表单为主，降低管理员和志愿者的学习成本，减少操作错误。

### 3.4.3 可维护性需求

系统应采用前后端分离架构，将接口、业务逻辑和页面展示分开处理；后端采用分层设计，将控制层、服务层和数据访问层分离，便于后续修改和扩展。数据库表和接口命名应保持统一，便于论文中说明和后续代码维护。

### 3.4.4 性能需求

系统主要面向社区管理场景，数据规模相对有限，因此不需要超高并发架构，但页面加载、列表查询和基本提交操作应保持较快响应。分页查询、数据库索引和接口返回结构应尽量保持简洁，以保证用户操作流畅。

### 3.4.5 兼容性需求

系统前端应能够在主流浏览器中正常显示和运行，后端接口应支持通过标准 HTTP 请求访问。前端使用 Vue 3 和 Vite 构建，有利于保持现代浏览器兼容性，并便于后续部署和演示。

## 3.5 可行性分析

### 3.5.1 技术可行性

本系统采用 Spring Boot、Vue 3、MySQL 和 JWT 等成熟技术实现，相关技术均有较多开发资料和实践案例，适合本科毕业设计项目。当前开发环境已经确认 Java 21 和 Node 22 可正常使用，后端与前端项目均已搭建完成并可以正常运行，因此从技术角度看具备较强可行性。

### 3.5.2 经济可行性

系统开发主要基于开源技术栈，Spring Boot、Vue、MySQL、MyBatis-Plus 和 Element Plus 均可免费使用，不需要额外采购商业软件或硬件设备。测试阶段使用本地环境和 Docker 容器即可完成，因此整体开发成本较低，符合毕业设计项目的经济条件。

### 3.5.3 操作可行性

系统面向社区志愿者服务管理场景，管理员和志愿者的操作流程清晰，页面设计也围绕实际业务场景展开。登录、列表、表单、弹窗、审核和签到等操作模式较为常见，用户容易理解和上手，因此从操作层面看具备较强可行性。

### 3.5.4 时间可行性

当前系统功能边界已经收敛为高频次、强业务联系的核心闭环，主要开发内容聚焦于权限认证、志愿者管理、活动生命周期管理及统计展示。由于架构选型成熟、功能边界明确且前后端并行实现路径清晰，能够在标准的研发周期内容量评估下按时交付测试与上线，具备较高的时间周期掌控性与可行性。

## 3.6 本章小结

本章结合社区志愿者服务管理的实际业务场景，对系统的角色需求、业务流程需求、系统非功能属性约束以及项目落地的综合可行性进行了系统性分析。通过需求分解可以看出，工程实践的目标并非盲目追求全场景的大而全，而是在聚焦核心业务脉络、遵循敏捷迭代原则的前提下，提供一个高可用、易扩展的信息化解决方案。需求分析的结果构成了下文系统总体与详细设计的核心业务依托。
